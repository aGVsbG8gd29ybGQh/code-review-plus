## ✨ 功能特性

### 1️⃣ 智能状态跟踪

为每个代码变更块标记评审状态，支持三种状态：

- 🔵 **未评审** - 待处理的代码变更
- ✅ **通过** - 评审无问题
- ⚠️ **问题** - 需要关注或修改

**自动级联更新**：代码块状态变更时，自动重新计算文件状态，保持数据一致性。

![状态跟踪示例](docs/images/status-tracking.png)
> 📸 **截图说明**：展示工具窗口中的文件树，包含不同状态的图标显示

---

### 2️⃣ 增量评审支持

分支更新后智能合并新旧差异，**保留已评审状态**，只需评审新增或变更的代码。

**智能匹配策略**：
- ✅ **精确匹配** - 代码位置和内容完全一致，保留评审状态
- 🔄 **模糊匹配** - 代码内容变化但可追溯，保留评论但重置状态
- ➕ **新增识别** - 自动识别新增代码块
- ➖ **删除归类** - 清理已删除的代码块

![增量评审示例](docs/images/incremental-review.png)
> 📸 **截图说明**：刷新会话前后的对比，展示已评审状态的保留效果

---

### 3️⃣ 持久化会话管理

评审进度自动保存到本地SQLite数据库，支持：

- 💾 **自动保存** - 每次状态变更实时持久化
- 🔄 **会话恢复** - IDEA重启后自动恢复上次活动会话
- 📁 **多项目支持** - 在同一IDEA实例中管理多个Git项目
- 📊 **多会话并行** - 同时管理多个评审分支

![会话管理示例](docs/images/conversation-manager.png)

### 6️⃣ 代码评论系统 🆕

为代码评审提供完整的评论能力，支持问题标记和追踪：

- 💬 **Diff 窗口添加评论** - 右键菜单为任意代码行添加评论
- 🎨 **可视化评论图标** - Gutter 区域显示状态化图标（待处理/无需处理/已处理）

**智能特性**：

- 通过 Git 历史回显评论创建时的代码内容
- 批量更新评论状态，高效管理评审进度

## 🚀 快速开始

### 第一步：打开工具窗口

通过以下任一方式打开工具窗口：

- **菜单栏**: `View` → `Tool Windows` → `Code Review Plus`
- **侧边栏**: 点击左侧工具栏的 `Code Review Plus` 图标

---

### 第二步：创建评审会话

1. 打开创建会话对话框
   - **菜单栏**: `Tools` → `Code Review Plus` → `Start Review...`
   - **项目右键菜单**: `Code Review Plus` → `Start Review...`
   - **工具窗口**: 点击工具栏`+`号
2. 在弹出对话框中选择：
   - **评审分支**：需要评审的Feature分支（例如：`feature/user-login`）
   - **基准分支**：对比的目标分支（例如：`master` 或 `develop`）
3. 点击 **"确定"** 创建会话

**提示**：
- 评审分支和基准分支必须已推送到远程仓库
- 系统会自动检查分支关系并给出警告

---

### 第三步：开始评审

1. **浏览文件列表**：在工具窗口中查看所有变更文件
2. **打开Diff视图**：双击文件名打开对比视图
3. **标记评审状态**：点击代码行号旁的图标切换状态
   - 🔵 未评审 → ✅ 通过 → ⚠️ 问题 → 🔵 未评审（循环）
4. **查看进度**：底部状态栏实时显示评审统计

---

### 第四步：添加评论（可选）🆕

为代码添加评论标记问题和建议：

1. **在 Diff 视图中右键选择"添加评论"**
2. **输入评论内容并保存**（最多 500 字符）
3. **点击评论图标查看和管理评论**
4. **使用评论列表视图集中管理**（工具栏"切换视图"）

💡 **提示**：详细使用方法请参见 [使用指南 - 评论系统使用](#评论系统使用-)

---

### 第五步：增量刷新

当评审分支有新提交时：

1. 点击工具栏的 **"刷新"** 按钮
2. 系统自动：
   - 计算新的diff
   - 保留已评审状态和评论
   - 标记新增/变更代码为"未评审"
3. 继续评审新增或变更的代码

## 🎯 适用场景

### ✅ 推荐场景

1. **Feature分支评审**
   - 合并到主分支前的代码评审
   - 确保代码质量符合团队标准

2. **大型分支评审**
   - 变更文件较多，需要分批次评审
   - 避免一次性评审导致遗漏

3. **持续更新分支**
   - 评审过程中分支持续更新
   - 利用增量评审避免重复劳动

4. **团队内部评审**
   - 个人评审后标记状态
   - 便于后续团队讨论和追溯

---

### ❌ 不推荐场景

1. **Pull Request平台**
   - 如果团队使用GitHub/GitLab的PR功能
   - 建议直接在平台上进行评审和讨论

2. **非Git项目**
   - 插件依赖Git仓库
   - SVN等其他版本控制系统不支持

3. **跨仓库对比**
   - 仅支持同一仓库内的分支对比
   - 不支持跨仓库的代码对比

## 🏗️ 技术架构

### 核心技术栈

- **开发语言**: Kotlin 1.9+
- **构建工具**: Gradle 8.5+ (Kotlin DSL)
- **平台SDK**: IntelliJ Platform SDK 2025.2.3
- **数据库**: SQLite JDBC
- **架构模式**: MVVM (ViewModel + UiState + Intent)

---

### 分层架构

```
┌─────────────────────────────────────┐
│      Presentation Layer (UI)       │  ← 工具窗口、对话框、Diff扩展
├─────────────────────────────────────┤
│      ViewModel (MVVM)               │  ← 状态管理、意图处理
├─────────────────────────────────────┤
│      Business Layer                 │  ← 业务逻辑、规则校验
├─────────────────────────────────────┤
│      Service Layer                  │  ← Git服务、匹配策略
├─────────────────────────────────────┤
│      Repository Layer               │  ← 数据访问抽象
├─────────────────────────────────────┤
│      Database Layer (SQLite)        │  ← 数据持久化
└─────────────────────────────────────┘
```

---

### 数据存储

- **存储位置**: `{projectPath}/.idea/code-review-plus.db`
- **数据库类型**: SQLite (WAL模式)
- **表结构**:
  - `conversations` - 评审会话表
  - `file_review_items` - 文件评审项表
  - `diff_blocks` - Diff代码块表
  - `comments` 🆕 - 评论数据表（存储评论内容、状态、关联关系）

**数据关联**：
```
conversation (会话)
  ├── file_review_item (文件)
  │     ├── diff_block (代码块)
  │     └── comment (评论) 🆕
  └── comment (评论可直接关联会话)
```

**隐私说明**: 所有数据仅存储在本地，不会上传到任何服务器。

## ❓ 常见问题

### Q1: 插件如何识别代码块是否已评审？

**A**: 插件为每个代码块计算唯一签名（基于代码位置和内容的MD5），刷新时通过签名匹配识别：
- **精确匹配**：位置和内容都一致 → 保留评审状态
- **模糊匹配**：内容有变化但可追溯（通过Git历史） → 重置为未评审
- **无法匹配**：视为新增或删除

---

### Q2: 刷新会话后，之前的评审状态和评论会丢失吗？

**A**: 不会！这正是插件的核心功能：

**评审状态保留**：
- ✅ 代码未变更 → 完全保留状态
- 🔄 代码有变化 → 重置为未评审（避免误导）
- ➕ 新增代码 → 标记为未评审
- ➖ 删除代码 → 自动清理

**评论保留机制** 🆕：
- ✅ 代码未变更 → 完全保留评论
- 🔄 代码有变化 → 保留评论，但标记为悬浮评论（半透明显示）
- 📝 悬浮评论仍可查看原始代码和评论内容
- 🗑️ 可手动删除不再需要的悬浮评论

---

### Q3: 是否支持多人协作评审？

**A**: 当前版本**不支持**多人协作。评审状态存储在本地数据库（`.idea/`目录），无法跨机器共享。

**建议**：
- 团队协作评审请使用GitHub/GitLab的PR功能
- 本插件更适合个人预评审或本地代码检查

---

### Q4: 插件对性能有影响吗？

**A**: 性能影响极小：
- **Diff计算**: 使用Git命令行，性能与IDEA原生Diff一致
- **数据库操作**: SQLite轻量级，读写速度快（< 5ms）
- **UI渲染**: 仅在打开工具窗口时加载，不影响编辑器性能

**优化措施**：
- 单文件刷新避免全量更新
- 批量数据库操作减少IO
- 异步加载文件树

---

### Q5: 是否支持为代码块添加评论？

**A**: **已支持**！🎉 插件提供完整的评论系统：

**核心功能**：
- 💬 **添加评论** - 在 Diff 视图中右键为代码块添加评论
- 🎨 **可视化图标** - Gutter 区域显示评论状态图标
- 📝 **评论管理** - 编辑评论内容、切换状态（待处理/无需处理/已处理）
- 🔄 **智能保留** - 刷新会话时自动保留评论，代码变更后标记为悬浮评论
- 📋 **列表视图** - 集中管理所有评论，支持筛选和批量操作
- 🎯 **精准定位** - 双击评论跳转到 Diff 窗口对应位置
- 🕰️ **原始代码回显** - 通过 Git 历史查看评论创建时的代码上下文

**使用场景**：
- 标记代码问题和改进建议
- 追踪评审过程中的待办事项
- 记录代码讨论要点

**注意**：评论内容为纯文本格式（不支持 Markdown），最多 500 字符。

---

### Q5.1: 评论功能有哪些限制？🆕

**A**: 评论功能当前限制：

**内容限制**：
- 📝 **纯文本格式**：不支持 Markdown 或富文本
- 📏 **最大长度**：500 字符
- 🚫 **不支持附件**：无法插入图片或文件

**协作限制**：
- 💾 **本地存储**：评论存储在本地数据库，不支持多人共享
- 🔒 **无法同步**：不支持导出或同步到其他机器
- 📤 **导出功能**：暂不支持导出评论报告（计划在后续版本添加）

**功能限制**：
- 🔗 **关联限制**：评论关联到 Diff Block，代码变更后变为悬浮评论
- 🔍 **搜索范围**：暂不支持搜索评论内容，只能搜索文件名

**最佳实践建议**：
- ✅ **个人评审**：使用评论记录问题和改进建议
- ✅ **本地检查**：作为提交前的本地代码自查工具
- ⚠️ **团队协作**：建议使用 GitHub/GitLab PR 的评论功能
- 🧹 **定期清理**：清理已处理的评论保持列表整洁

### Q8: 为什么创建会话时提示"分支未推送"？

**A**: 插件要求评审分支和基准分支都已推送到远程仓库，原因：
- 确保评审的是最新代码
- 避免本地未提交代码干扰diff计算
- 保证团队成员看到相同的代码版本

**解决方法**：
```bash
# 推送评审分支
git push origin feature/your-branch

# 如果基准分支也未推送
git push origin master
```

### Q12: 评论和评审状态有什么区别？ 🆕

**A**: 两者针对不同的对象和用途：

**评审状态**（Diff Block 级别）：
- 🎯 **作用对象**：整个代码变更块（Diff Block）
- 📊 **状态类型**：未评审、通过、问题
- 🔄 **状态流转**：未评审 → 通过/问题
- 📈 **用途**：追踪整体评审进度

**评论状态**（Comment 级别）：
- 🎯 **作用对象**：具体的评论记录
- 📝 **状态类型**：待处理、无需处理、已处理
- 🔄 **状态流转**：待处理 → 无需处理/已处理
- 💬 **用途**：追踪具体问题的处理进度

**典型使用场景**：
1. 标记 Diff Block 为"问题"（表示这块代码有问题）
2. 为该 Diff Block 添加评论，详细说明问题（状态：待处理）
3. 开发者修复后，更新评论状态为"已处理"
4. 确认修复后，更新 Diff Block 状态为"通过"

---

### Q13: 悬浮评论是什么？如何产生的？ 🆕

**A**: **悬浮评论**是指关联的代码块（Diff Block）已不存在，但评论仍保留的情况。

**产生场景**：
- 🔄 **分支更新**：刷新会话后，原代码块被修改或删除
- 🔀 **代码合并**：评论的代码已合并到目标分支
- ⏮️ **代码回滚**：评论的代码被回滚或撤销

**视觉标识**：
- 🌫️ Diff 窗口中评论图标显示为**半透明**
- 📋 评论列表中文本颜色为**灰色**

**实际意义**：
- ✅ 保留评审历史记录，不因代码变更丢失
- 📝 可查看评论创建时的原始代码（通过 Git 历史）
- 🗑️ 可手动删除不再需要的悬浮评论

**建议**：
- 定期清理已处理的悬浮评论
- 保留有价值的历史讨论记录

### Q14: 如何高效管理大量评论？ 🆕

**A**: 插件提供多种方式帮助您管理评论：

**1. 使用评论列表视图**：
- 点击工具栏的"切换视图"按钮
- 选择"评论列表"集中查看所有评论
- 支持按文件分组展示

**2. 状态筛选**：
- 使用筛选功能只显示特定状态的评论
- 快速定位"待处理"评论
- 隐藏已完成的评论

**3. 批量操作**：
- 选中多个评论（Ctrl/Cmd + 点击）
- 右键菜单批量更新状态
- 提升处理效率

**4. 搜索定位**：
- 使用搜索框快速定位文件
- 双击评论跳转到 Diff 窗口对应位置

**5. 定期清理**：
- 删除已处理且不再需要的评论
- 清理悬浮评论（代码已变更）
- 保持评论列表整洁

**最佳实践**：
- 📝 评论描述简洁明确（最多 500 字符）
- 🏷️ 及时更新评论状态
- 🗑️ 定期清理已完成的评论
- 📊 配合评审状态使用，形成完整的评审流程
